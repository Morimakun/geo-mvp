## 📋 帳票の基本構造

### ブロック分割

```
┌─────────────────────────────────────────────────────────────────┐
│ 上部情報: 日付 | 店舗名 | 氏名                                    │
├──────────────┬──────────────────┬──────────────────────────────┤
│              │                  │                              │
│  左ブロック   │   中央ブロック    │      右ブロック              │
│  (項目1-25)  │  (項目26-43)    │      (項目44-79)             │
│              │  (項目80-101)   │                              │
│              │                  │                              │
├──────────────┴──────────────────┴──────────────────────────────┤
│ 下部情報: 項目102(日報データNo) | 項目103(タブレットNo)          │
└─────────────────────────────────────────────────────────────────┘
```

### 各ブロックの構成

#### **左ブロック（項目1～25）**

| 項目 | 内容 | 備考 |
|-----|------|------|
| 1-4 | 基本情報（各サービス新規申込） | 縦2列×複数行 |
| 5 | **左側合計1** | 重要：合計欄 |
| 6-11 | オプション申込数 | 縦2列 |
| 12-18 | 固定電話・デジタル系 | 縦2列 |
| 19-24 | 各種サービス数 | 縦2列 |
| 25 | **左側合計2** | 重要：合計欄（合計行） |

#### **中央ブロック（項目26～43、80～101）**

| 範囲 | 内容 |
|-----|------|
| 26-43 | スタッフ数、ユーザー登録数、NETFLIX等のサービス利用状況 |
| 80-89 | 顧客年齢別データ（20代～90代以上） |
| 90-91 | イベント参加数、UQスポット手数 |
| 92-101 | キャンペーン数、クーポン数、購入人数 etc. |

#### **右ブロック（項目44～79）**

| 項目 | 内容 |
|-----|------|
| 44-78 | 各種サービス数（ネット、固定電話、テレビ、パッケージ等） |
| 79 | **右側合計** | 重要：合計欄 |

## 📖 読み取り方針

### 1. 固定位置読み取り（日付・店舗・氏名・キー）

```python
# 上部固定項目
crop_date = crop_top_left(page_image)        # 日付
crop_store = crop_top_center(page_image)     # 店舗名
crop_name = crop_top_right(page_image)       # 氏名

# 下部固定項目
crop_data_no = crop_bottom_left(page_image)   # 日報データNo
crop_tab_no = crop_bottom_right(page_image)   # タブレットNo
```

### 2. 合計欄読み取り（Crop + 専用プロンプト）

#### **左側合計欄（項目5、25）**

```python
# 項目5の左側合計1をcrop
crop_left_total_1 = crop_item_5(page_image)

# 項目25の左側合計2をcrop（3列すべて取得）
crop_left_total_2 = crop_item_25(page_image)

# 読み取りプロンプト（例）
prompt = """この画像は帳票の合計欄です。
黒く潰れた見出しは無視してください。
手書きで記入された数字のみを左から右に読み取ってください。
数字が不明瞭な場合は候補を複数記録してください。"""
```

#### **右側合計欄（項目79）**

```python
# 項目79の右側合計をcrop
crop_right_total = crop_item_79(page_image)

# 読み取り
# 同様に手書き数字のみを抽出
```

### 3. 手書き数字の判別精度対応

#### **出力形式**

```json
{
  "item_number": 5,
  "location": "left_total_1",
  "extracted_values": ["18", "5", "2", "2", "0", "0", "0"],
  "candidates": {
    "index_2": ["2", "3"],
    "index_3": ["2", "3", "5"]
  },
  "confidence": "medium",
  "needs_review": true,
  "review_reason": "手書き数字の2/3判別が不明瞭。正の字との整合確認が必要です。"
}
```

#### **判別が難しい数字**

| パターン | 対応 |
|---------|------|
| **2 vs 3** | 上部の線の有無で判定が分かれる |
| **1 vs 7** | 上部の横線の有無 |
| **0 vs 6** | 閉じ具合 |
| **5 vs 8** | 中央部の突出度 |

→ **候補として複数保持、needs_review=true**

### 4. 正の字・内訳との突き合わせ

#### **検証ロジック**

```
正の字の棒の数 → 推定値
合計欄の数字 → 抽出値
内訳の各項目の合計 → 検算値

→ 3つの値が一致するか確認
→ 不一致の場合は needs_review=true
```

#### **例**

```
正の字: |||||| (6本) → 推定値=6
合計欄: "6" → 抽出値=6
内訳合計: 1+2+1+1+1 = 6 → 検算値=6

結果: 一致 ✓ needs_review=false
```

```
正の字: ||||||| (7本) → 推定値=7
合計欄: "2"または"3" → 抽出値=候補
内訳合計: 1+2+1+1+2 = 7 → 検算値=7

結果: 不一致 ✗ needs_review=true
```

## 🔍 読み取り順序と優先度

### **優先度1：照合キー（必須）**
1. 日報データNo（主キー）
2. タブレットNo（副キー）
3. 日付
4. 店舗名

### **優先度2：検証用データ**
5. 氏名
6. 左側合計欄（項目5, 25）
7. 右側合計欄（項目79）
8. 件数（正の字の棒）

### **優先度3：補助情報**
9. 各項目の内訳数字
10. 備考欄（事務局への連絡）

## 📊 CSV突合設計

### **主キー戦略**

```
【第一優先】日報データNo
  - 最も固定値
  - 見出し直下の固定位置
  - crop読み取りで高精度

【第二優先】タブレットNo
  - フォールバック用
  - 複数帳票でのNo重複時の判定補助

【第三優先】日付＋店舗名＋氏名
  - 日報データNo未取得時のフォールバック
  - ただし「同一者が複数回投稿」の場合は曖昧性あり
```

### **補助チェック項目**

```
検証項目                    | 突合方法
─────────────────────────┼──────────────
件数（正の字）              | 左側合計と一致確認
項目5合計                  | 内訳合計と比較
項目25合計                 | 内訳合計と比較
項目79合計                 | サービス別内訳と比較
正の字/内訳                | 合計欄と三角チェック
```

### **一致判定フロー**

```
【Step 1】日報データNo で検索
  ├─ 完全一致 → そのレコードと突合
  ├─ 複数一致 → 【Step 2】へ
  └─ 未取得 → 【Step 3】へ

【Step 2】タブレットNo で絞り込み
  ├─ 一意決定 → そのレコードと突合
  └─ 複数一致 → needs_review=true

【Step 3】日付＋店舗名＋氏名 で検索
  ├─ 完全一致 → そのレコードと突合
  ├─ 複数一致 → needs_review=true
  └─ 未発見 → needs_review=true（新規？）

【Step 4】補助チェック
  - 件数が一致するか
  - 合計欄が整合するか
  - 不整合 → needs_review=true に昇格
```

## ⚠️ 実装時の注意事項

### **座標設定の柔軟性**

```
このクリア版は「理想状態」です。
実FAXでは以下の差異が発生します：

・傾き: ±3～5度
・ズレ: ±5～10mm（スキャン時）
・黒つぶれ: 複写機の濃度ムラ
・縮尺差: スキャン解像度、原稿のズレ
```

### **Cropの安全マージン設定**

```python
# 固定値ではなく、余白を含めたcrop
left_margin = width * 0.02   # 左右2%の余白
top_margin = height * 0.02   # 上下2%の余白

crop_area = {
  'left': item_start_x - left_margin,
  'top': item_start_y - top_margin,
  'right': item_end_x + left_margin,
  'bottom': item_end_y + top_margin
}
```

### **読み取り失敗時の対応**

```
❌ 自動で空文字列や0を埋める
❌ 他の帳票の数字で推測する
❌ 確認なしで一致判定する

✅ 読み取り失敗 → needs_review=true
✅ 候補複数 → candidates に記録
✅ 不整合 → review_reason に理由を記載
```

### **手書き判別の慎重さ**

```
例：左側合計25 の第2列が "3" に見える場合

✓ 正しい対応：
  extracted_values: ["3", "3", "3"]
  candidates: {1: ["2", "3"]}
  confidence: "medium"
  needs_review: true
  review_reason: "第2列の2/3判別が不明瞭"

✗ 不正な対応：
  extracted_values: ["3", "3", "3"]
  confidence: "high"
  needs_review: false
```
